# 化石文件:我的。Emacs

> 原文:[https://hackaday.com/2022/12/10/fossil-files-my-emacs/](https://hackaday.com/2022/12/10/fossil-files-my-emacs/)

上周，我[在一个更广泛的背景下写了关于货物养殖的文章](https://hackaday.com/2022/12/03/cargo-culting-and-buried-treasure/)，所以这个星期我将全盘托出。让我思考这个话题的文件是你可能见过的最糟糕的情况:我有一个`.emacs`文件，自从 1994 年在计算机实验室*从别人那里复制过来之后，我就没有真正理解过它——可能是 Ben Scarlet，他的名字就刻在那里！是的，我的`.emacs`档都快 30 了，还不是很懂，不完全懂。*

现在我为自己辩护，几年前我切换到`vim`作为我的主要编辑，但是这一个文件已经在运行 1.0 之前版本的 Linux 的 Pentiums 上，在前面提到的计算机实验室的 IBM RS/6000 机器上，以及在我多年来拥有的一系列笔记本电脑和台式机上工作过。它让我读完了本科，研究生，并工作了十年。这对我很有好处。如果我现在启动`emacs`，它还会在这里。

对于那些不使用`emacs`的人来说，`.emacs`文件是一个配置文件。它说明了如何根据扩展名解释不同的文件，定义了一些特殊的组合键，也许最重要的是，定义了代码语法突出显示的工作方式。这基本上是`emacs`中所有*特有的外观和感觉，也是我的`emacs`属于我的原因。但是我不明白。*

为什么？因为它是用 LISP 编写的，为了 GNU 的缘故，也因为它引用了所有`emacs`在幕后使用的神秘内部变量。我绝对不是说我没有调整周围的一些颜色，或在这里或那里胡乱修补一些东西，但程度总是限于我能逃脱的任何东西，而不必真正学习 LISP。

这个古老的文件化石证明了两件事。`emacs`代码库足够稳定，以至于在这么长时间后仍然可以工作，但是`emacs`是如此的复杂，并且是用一种晦涩难懂的语言编写的，以至于我从来没有花时间去真正地钻研它——障碍太高，付出的回报太低。我毫不怀疑我*能够*真正弄明白，但我就是没有。

所以我只是带着这个文件，从一台电脑到另一台电脑，没有理解它，也没有特别想理解它。除了现在我写这个。该死的。
